分析当前分支与 main 分支的差异，告诉开发者在合并代码之前应该测试哪些功能点。

## 分析步骤

1. 使用 `git diff main...HEAD --stat` 获取改动的文件列表
2. 使用 `git diff main...HEAD` 获取具体改动内容
3. 分析每个改动文件的影响范围

## 输出格式

### 改动概述
简要说明本次改动涉及的模块和主要变更。

### 需要测试的功能点
根据代码改动，列出开发者需要手动测试的具体功能点，格式如下：

- [ ] 功能点描述（说明为什么需要测试）

### 测试建议
如有必要，提供额外的测试建议或注意事项。

## 分析维度

根据改动的文件路径，重点关注以下影响：

- `internal/services/*`：对应服务的 API 功能
- `internal/services/windows/*`：窗口的创建、显示、隐藏、关闭行为
- `internal/services/i18n/*`：语言切换、文案显示
- `internal/sqlite/*`：数据库初始化、迁移、数据读写
- `internal/bootstrap/*`：应用启动流程
- `frontend/src/*`：页面渲染、用户交互、前后端通信
- `pkg/*`：公共包功能，注意跨平台兼容性

## 注意事项

- 只列出与本次改动相关的测试点，不要列出无关的通用测试项
- 测试点要具体、可操作，避免笼统描述
- 如果改动涉及多个模块的交互，需要特别指出集成测试点
